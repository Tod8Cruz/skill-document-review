# 문서 검토 스킬 (document-review)

> 여러 전문가 페르소나가 참여하는 **가상 검토 회의**를 열어, 하나의 문서를 다관점에서 비평하고
> 실행 가능한 개선안과 개선 문서(V2)까지 산출하는 Claude Code 스킬.

단일 관점으로는 놓치기 쉬운 문제를, 서로 보완하면서도 일부는 충돌하는 검토자 4~5명의 시점으로
드러낸다. 검토는 의장이 중재하는 2라운드 회의로 진행되며, 모든 과정이 회의록으로 실시간 기록된다.

---

## 이런 게 필요할 때

기획서·PRD, 제안서·사업계획서, 보고서, 계약서, 마케팅 카피, 기술 스펙/설계 문서, 정책/프로세스
문서 등 **여러 이해관계자의 시선이 필요한 문서**에 대해 깊이 있는 피드백을 받고 싶을 때 사용한다.

다음과 같이 말하면 트리거된다:

- "이 문서 검토해줘 / 리뷰해줘"
- "검토 회의 열어줘 / 다관점 리뷰"
- "문서 피드백 줘"
- "review this document" / "document review"

## 빠른 시작

```
/document-review <검토할 문서 경로>
```

예: `/document-review docs/prd.md`

경로를 주지 않아도 다음 우선순위로 대상 문서를 찾는다:

1. 스킬 인자로 받은 경로
2. IDE에서 열어둔 / 선택한 파일
3. 둘 다 없으면 검토할 문서 경로를 직접 묻는다

회의·산출물의 **언어는 원본 문서의 언어를 따른다** (한국어 문서 → 한국어 회의).

## 동작 방식

당신(메인 컨텍스트)은 회의의 **의장**이고, 실제 검토 의견은 모두 페르소나 서브에이전트가 낸다.
의장은 진행·취합·합의·기록만 하며 내용에 대한 본인 의견은 내지 않는다.

```
Step 0  대상 문서 확보            문서 전체를 읽고 유형·목적·독자·핵심 주장 파악
Step 1  참석자 도출 → 페르소나 생성  문서 맞춤 검토자 4~5명을 새로 만들어 파일로 저장
Step 2  사용자 검토·승인           페르소나 구성을 보여주고 AskUserQuestion으로 승인
Step 3  검토 회의                개회 → R1 의견발표 → R2 반박·첨언 → 합의 도출 → 폐회
Step 4  산출물 생성              회의록 / recommendations / V2 문서
```

각 Step의 핵심 결과(페르소나 목록, 라운드별 발언, 합의안)는 사용자 화면에도 실시간으로
보여줘 — 회의가 눈앞에서 진행되는 것처럼.

### 핵심 설계 원칙

- **페르소나는 매번 새로** — 고정 아키타입에서 고르지 않고, 이 문서를 직접 읽어 꼭 필요한
  참석자를 도출해 새 인물로 만든다. 이전 검토의 페르소나를 재사용하지 않는다.
- **페르소나 ≠ 검토 결론** — 페르소나 파일은 "이 사람이 누구인가"(경력·전문성·기질·가치관·
  말투)만 담는다. "이 문서의 무엇이 문제다"는 절대 미리 적지 않고, 회의에서 인물이 직접 만든다.
- **Round 1은 병렬 독립** — 서로의 의견을 모른 채 동시에 시작해 앵커링을 막는다.
- **원문 직접 읽기** — 의장이 문서를 요약·발췌해 넘기지 않는다. 각 에이전트가 원본 파일과 자기
  페르소나 파일을 **경로로 직접 `Read`** 한다.
- **갈등을 살린다** — 의견 충돌은 정상이자 가치다. 억지로 봉합하지 않고, 합의 못 한 것은
  "소수의견"으로 정직하게 남긴다.
- **원본 불가침** — 원본(V1)은 어떤 경우에도 수정·삭제하지 않는다. 개선본은 항상 별도의 V2 파일.
- **실시간 기록** — 회의 시작 시 회의록 파일을 먼저 만들고 발언마다 즉시 append한다. 중간에
  세션이 끊겨도 그때까지의 회의록이 파일에 남는다.

## 산출물

스킬은 원본 문서가 있는 폴더 안에 다음을 만든다:

```
<문서 폴더>/
├── review-personas/                       # 이번 검토용 페르소나 (매번 새로)
│   ├── 1-<이름>.md
│   ├── 2-<이름>.md
│   └── … (총 4~5개)
└── <원본명>-검토-<YYYYMMDD-HHMM>/          # 회의 1회 = 폴더 1개 (타임스탬프)
    ├── 회의록-transcript.md                # 전체 회의록 (실시간 작성)
    ├── recommendations.md                  # 우선순위별 추천안
    └── <원본명>-v2.<확장자>                 # 개선 문서 (변경이력 포함)
```

| 산출물 | 내용 |
|--------|------|
| `review-personas/<번호>-<이름>.md` | 검토자 인물 브리프. 경력·전문성·기질·가치관·검토 렌즈·말투. 문서 평가는 들어있지 않음 |
| `회의록-transcript.md` | 개회 → Round 1(독립 의견) → Round 2(반박·첨언) → 합의 → 폐회 전 과정 발언록 |
| `recommendations.md` | 합의된 추천안을 우선순위순으로. 각 항목: 이슈 · 추천 · 우선순위 · 제기한 페르소나 · 합의 수준 · 근거 |
| `<원본명>-v2.<확장자>` | 합의·다수 추천을 반영한 개선본. 상단에 변경이력, 하단에 "미반영 의견" 섹션 |

**표기 규칙**

- **우선순위**: `High` / `Medium` / `Low`
- **합의 수준**: `만장일치` / `다수` / `소수의견`
- 소수의견은 V2 본문에 반영하지 않고, V2 하단 "미반영 의견"과 `recommendations.md`에만 기록한다.
- `<원본명>-v2`가 이미 있으면 `-v3`, `-v4`로 번호를 올린다.

## 실제 예시 (Worked Example)

이 스킬을 `docs/prd.md`(소셜 라운지 관련 PRD)에 실제로 돌린 산출물이
이 저장소 `docs/`에 그대로 남아 있다.

```
docs/
├── prd.md                                 # 원본 (검토 대상)
├── review-personas/
│   ├── 1-이재혁.md      # 프로덕트 매니저 — MVP 범위·우선순위
│   ├── 2-강도현.md      # 보드게임 놀이 교육자 — 경험·관계·놀이 정체성
│   ├── 3-한소영.md      # 커뮤니티 운영 매니저 — 운영 현실성·정책
│   ├── 4-윤지아.md      # UX 디자이너 — 사용자 여정·접근성
│   └── 5-오선희.md      # 55세 실사용자 — 신뢰·명료함·환영받는 느낌
├── prd-검토-20260623-2059/
│   ├── 회의록-transcript.md                # 5인 2라운드 회의 전문
│   └── recommendations.md                  # R1~R11 우선순위별 추천안
└── prd-v2.md                              # 개선본 (변경이력 v1→v2 포함)
```

> 참고: 현재 스킬 사양은 V2 문서도 타임스탬프 폴더 안에 두지만, 이 예시는 V2를 `docs/`
> 최상위(`prd-v2.md`)에 둔 초기 실행 결과다. 위치는 조금 다를 수 있다.

**흐름이 어떻게 이어지는지** — 회의에서 도출된 추천안이 R 코드로 추적되어 V2에 반영된다:

- **R1**: 라운지를 "1시간 독점 예약" → "드롭인/정원 입장"으로 재정의 (강도현 제기 → 만장일치)
- **R2**: 결제 금액을 공개하고, 등급 선택을 가입 이후로 미룸 (오선희·윤지아)
- **R3**: 랜딩·프로그램·피드에 "놀이/관계" 정체성을 명시적으로 노출 (강도현)
- … R4~R11까지 `recommendations.md`에 정리되고, 각 항목이 `prd-v2.md` 상단 변경이력에 매핑됨

즉 **회의록(왜)** → **recommendations(무엇을)** → **V2(어떻게 반영)** 가 하나로 연결된다.

## 페르소나 이해하기

페르소나 파일은 **"어떤 사람인가"에만 집중**한다. 파일만 읽으면 "이 사람은 어떤 전문가다"는
또렷이 떠오르되, "이 문서의 검토 결론"은 전혀 드러나지 않아야 한다. 각 파일의 구성:

```markdown
# <이름> — <역할/직함>
> <한 줄 인물 소개>
## 경력·배경
## 전문성
## 기질·성향          # 비판적/지지적/실용적/비전형/디테일형 中 1~2개 + 구체 묘사
## 직업적 가치관       # 도메인 일반론 — 특정 문서와 무관
## 검토 렌즈와 회의 태도
```

**다양성 규칙**

- 기질을 분산한다 — 최소 1명 지지적, 1명 비판적, 나머지는 실용·비전·디테일.
- 직업적 가치관·검토 렌즈가 서로 **부딪히도록** 배치한다 (예: 비용 중시 vs UX 중시). 가치관이
  충돌해야 Round 2 반박이 살아난다.
- **결정권자**(문서의 운명을 좌우할 사람)를 최소 1명 포함한다.

## 커스터마이징

- **페르소나 승인 루프 (Step 2)** — 회의 전, 페르소나 구성을 표로 보여주고 `AskUserQuestion`으로
  승인을 받는다. "이대로 진행 / 특정 페르소나 수정 / 페르소나 교체·추가" 중 선택할 수 있고,
  수정 요청 시 `review-personas/` 파일까지 갱신해 다시 보여준다. **승인 전에는 회의를 시작하지
  않는다.**
- **`references/persona-archetypes.md`** — 문서 유형별 검토자 체크리스트. 빠뜨린 관점이 없는지
  점검하는 **느슨한 참고용**이며, 여기 적힌 역할을 그대로 데려오지 않는다.
- **`references/output-templates.md`** — 회의록 / recommendations / V2 산출물의 정확한 포맷.
- **언어** — 원본 문서 언어를 따른다. 영어 문서면 회의도 영어로 진행된다.

## 사용 도구 / 권한

- `allowed-tools`: `Read, Write, Bash, Glob, Agent, AskUserQuestion`
- 페르소나 발언은 `general-purpose` 서브에이전트가 담당한다 (파일을 경로로 직접 `Read`).
  - Round 1: 병렬 spawn (앵커링 방지)
  - Round 2: `SendMessage`로 같은 에이전트를 이어가며 진행 (불가하면 새 에이전트 spawn 폴백)
- `Bash`는 타임스탬프 생성(`date +%Y%m%d-%H%M`)과 폴더 준비에, `Glob`은 기존 V2 버전 확인에 쓰인다.
- 버전: `1.0.0`

## FAQ

**페르소나를 재사용하나요?**
아니요. 매 검토마다 그 문서에 맞는 인물을 새로 도출해 만듭니다. `review-personas/`에 기존
파일이 있으면 모두 지우고 새로 채웁니다.

**페르소나끼리 의견이 갈리면요?**
정상입니다. 합의된 것은 만장일치/다수로, 합의 못 한 것은 소수의견으로 정직하게 기록합니다.
의장은 무한 토론으로 끌지 않고, 미해결 쟁점은 1회만 절충 확인 후 마무리합니다.

**원본이 수정되나요?**
아니요. 원본(V1)은 절대 건드리지 않습니다. 개선안은 항상 `-v2`(이미 있으면 `-v3`, `-v4`)
별도 파일로 만듭니다.

**좋은 결과를 얻으려면?**
- 검토 대상 문서가 구체적일수록(섹션·근거·가정이 명시될수록) 지적도 구체적입니다.
- Step 2 승인 단계에서 기질·가치관이 충분히 다양하고 충돌하는지 확인하세요. 전원 비판적이거나
  전원 우호적이면 회의가 단조로워집니다. 필요하면 페르소나를 교체·추가하세요.
