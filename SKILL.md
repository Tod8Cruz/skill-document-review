# 문서 검토 스킬 (Document Review)

## 목적

한 문서를 **여러 독립적인 전문가 시점**에서 비평해, 단일 관점으로는 놓치는
문제를 드러낸다. 맥락에 맞는 페르소나 4~5명을 만들고, 의장이 중재하는 가상
검토 회의를 거쳐, 실행 가능한 recommendations와 그것이 반영된 개선 문서(V2)를
산출한다.

당신(메인 컨텍스트)은 이 회의의 **의장**이다. 의장은 회의를 진행·취합·합의·
기록하며, **내용에 대한 본인 의견은 내지 않는다**. 실제 검토 의견은 모두
페르소나 서브에이전트가 낸다.

## 전체 흐름

```
Step 0  대상 문서 확보
Step 1  참석자 도출 → 페르소나 신규 작성 → review-personas/ 폴더에 파일 저장
Step 2  페르소나 사용자 검토·승인  ← AskUserQuestion 루프
Step 3  검토 회의 — 개회 → R1 의견발표 → R2 반박·첨언 → 합의 도출 → 폐회
Step 4  산출물 생성 — 회의록 / recommendations / V2 문서
```

진행 중 각 Step의 핵심 결과(페르소나 목록, 각 라운드 발언, 합의안)는 사용자
화면에도 보여준다 — 회의가 눈앞에서 진행되는 것처럼.

---

## Step 0 — 대상 문서 확보

대상 문서를 다음 우선순위로 결정한다:
1. 스킬 인자로 받은 경로 (`/document-review <경로>`)
2. IDE에서 사용자가 선택한 영역 / 열어둔 파일
3. 위 둘 다 없으면 사용자에게 "검토할 문서의 경로를 알려달라"고 질문

문서를 **전체** 읽고 다음을 파악한다:
- **문서 유형**: 기획서/PRD · 제안서/사업계획서 · 보고서 · 계약서 ·
  마케팅 카피 · 기술 스펙/설계 문서 · 정책/프로세스 문서 · 기타
- **목적**: 이 문서가 달성하려는 것 (의사결정 설득? 정보 전달? 합의?)
- **대상 독자**: 누가 읽는가
- **핵심 주장·구조**: 문서가 내세우는 주요 포인트

문서가 너무 길어도 **의장이 요약·발췌해서 에이전트에게 넘기지 않는다.** Step 3에서
각 페르소나 에이전트가 원문 파일을 직접 `Read`로 읽는다. 의장이 여기서 파악한 핵심
섹션은 개회사·합의 정리용 참고일 뿐이다.

---

## Step 1 — 참석자 도출 및 페르소나 신규 생성

페르소나는 **고정 아키타입 목록에서 고르는 것이 아니다.** 매 검토마다 이
문서를 직접 읽고, 이 문서에 **꼭 필요한 참석자를 새로 도출**해 새 페르소나를
만든다. 같은 "기획서"라도 B2B SaaS와 학원 운영은 전혀 다른 참석자가 필요하다.

### 1-1. 참석자 도출

Step 0에서 파악한 문서 유형·목적·대상 독자·핵심 주장을 근거로, **"이 문서를
제대로 검토하려면 어떤 사람이 회의실에 있어야 하는가"** 를 먼저 자유롭게
나열한다. 아키타입 목록을 베끼지 말고 이 문서에서 도출한다:

- 이 문서가 영향을 주는 **이해관계자**는 누구인가
- 이 문서의 가장 큰 **리스크 영역**(기술·법무·재무·UX·운영 등)은 어디인가
- 이 문서로 무언가를 **결정할 사람**은 누구인가
- 이 문서의 주장에 **반대할 만한 사람**은 누구인가

그 결과를 "필요 참석자 후보" 목록으로 정리한다 (각 후보에 한 줄 근거).
이 중 **상호 보완적이면서 일부는 충돌하는** 4~5명을 최종 선정한다.

> `references/persona-archetypes.md`는 빠뜨린 관점이 없는지 점검하는
> **느슨한 체크리스트로만** 참고한다. 거기 적힌 역할을 그대로 데려오지 말 것.

### 1-2. 페르소나 신규 작성

선정한 참석자마다 **이번 검토 전용으로 새 페르소나**를 만든다. 이름·배경
설정은 매번 새로 짓는다 (이전 검토의 페르소나를 재사용하지 않는다).

> ⚠️ **페르소나 작성의 핵심 원칙 — "어떤 사람인가"에만 집중한다.**
> 페르소나는 **인물 그 자체**를 묘사하는 것이다. 이 사람의 경력·전문성·
> 기질·가치관·말투를 풍부하고 구체적으로 그린다. **대상 문서를 어떻게
> 평가할지, 무엇이 문제일지, 무엇을 지적할지는 절대 미리 쓰지 않는다** —
> 그건 Step 3 회의에서 이 인물이 직접 발언으로 만들어 낸다. 페르소나 파일만
> 읽었을 때 "이 사람은 어떤 전문가다"는 또렷이 떠오르되, "이 문서의 검토
> 결론"은 전혀 드러나지 않아야 한다.

각 페르소나는 다음 필드를 갖는다 (모두 **인물 묘사** — 특정 문서와 무관):

| 필드 | 설명 |
|------|------|
| 이름 | 실제 사람 이름 — 회의 발언자 식별용. 매번 새로 |
| 역할/직함 | 예: "시니어 백엔드 엔지니어", "타겟 고객 (30대 학부모)" |
| 경력·배경 | 어떤 길을 걸어온 사람인가 — 구체적·서사적으로 |
| 전문성 | 무엇을 깊이 아는가, 무엇으로 인정받는가 |
| 기질·성향 | 어떻게 사고하고 행동하는가. 비판적/지지적/실용적/비전형/디테일형 中 1~2개 + 구체 묘사 |
| 직업적 가치관 | 일에서 무엇을 중요하게 여기는가 — 도메인 일반론, 특정 문서와 무관 |
| 검토 렌즈 | 어떤 일반적 시각으로 문서를 보는가 (예: "보안 관점") — 구체적 발견이 아님 |
| 말투·회의 태도 | 회의에서 어떻게 말하는가 |

**다양성 규칙**:
- 기질을 분산시킨다. 전원 "비판적"이면 회의가 단조롭다. 최소 1명은 지지적,
  1명은 비판적, 나머지는 실용·비전·디테일로 섞는다.
- 직업적 가치관·검토 렌즈가 서로 **부딪히도록** 인물을 배치한다 (예: 비용을
  중시하는 사람 vs 사용자 경험을 중시하는 사람). 가치관이 충돌해야 Round 2
  반박이 살아난다. 단, 이것은 *인물의 가치관* 충돌이지 미리 정해 둔
  *문서 결론*의 충돌이 아니다.
- 결정권자(문서의 운명을 좌우할 사람)를 최소 1명 포함한다.

### 1-3. 페르소나를 파일로 저장

원본 문서가 있는 폴더 안에 `review-personas/` 폴더를 만든다
(`<원본폴더>/review-personas/`). 이미 있으면 이전 검토의 페르소나 파일을
**모두 지우고** 새로 채운다 (매번 새 페르소나 원칙).

페르소나 **한 명당 파일 하나**를 만든다:
`review-personas/<번호>-<이름>.md` (예: `1-김미정.md`).

각 파일의 형식 (**인물 묘사만** — 대상 문서 평가는 넣지 않는다):

```markdown
# <이름> — <역할/직함>

> <한 줄 인물 소개>

## 경력·배경
<이 사람이 걸어온 길. 구체적이고 서사적으로>

## 전문성
<무엇을 깊이 아는가, 무엇으로 인정받는가>

## 기질·성향
<어떻게 사고하고 행동하는가. 성향 키워드 + 구체 묘사>

## 직업적 가치관
<일에서 무엇을 중요하게 여기는가 — 도메인 일반론, 특정 문서와 무관>

## 검토 렌즈와 회의 태도
<어떤 일반적 시각으로 문서를 보는가, 회의에서 어떻게 말하고 부딪히는가>
```

> ⚠️ 이 파일에는 **대상 문서에 대한 구체적 검토 의견·발견·지적·결론을
> 적지 않는다.** 오로지 "이 사람은 누구인가"만 담는다. 문서에 대한 평가는
> Step 3 회의에서 비로소 생성된다.

이 파일들은 Step 3에서 서브에이전트에게 전달할 **페르소나 브리프의 원본**이
된다.

### 1-4. 의장 역할 명시

**의장**의 역할을 명시한다: 중립적 진행자. 발언 순서 관리, 의견 취합,
쟁점 정리, 합의 유도, 회의록 작성만 담당. 내용 의견 없음.

---

## Step 2 — 페르소나 사용자 검토·승인

먼저 "필요 참석자 후보" 도출 결과를 간단히 보여주고, 최종 선정한 페르소나를
표로 명확히 제시한다 (이름·역할·관점·관심사·성향). `review-personas/` 폴더에
파일을 저장했음을 알린다.

그다음 `AskUserQuestion`으로 승인을 받는다:
- 질문: "이 페르소나 구성으로 검토 회의를 진행할까요?"
- 선택지:
  - "이대로 진행" (추천)
  - "특정 페르소나 수정"
  - "페르소나 교체·추가"

수정·교체·추가 요청 시 반영하고, **`review-personas/` 폴더의 파일도 함께
갱신**한 뒤 **다시 제시·확인**한다. 사용자가 승인할 때까지 반복한다. 회의는
승인 전에 절대 시작하지 않는다.

---

## Step 3 — 검토 회의 (서브에이전트 오케스트레이션)

의장으로서 회의를 진행한다. 각 단계 발언은 **두 곳에 동시에 출력**한다:
1. **사용자 화면** — 회의가 눈앞에서 진행되는 것처럼 보여줌
2. **`transcript.md` 파일** — 각 발언이 나오는 즉시 append. 회의 중간에
   세션이 끊겨도 그때까지의 회의록은 파일로 남는다.

### 3-0. 산출 폴더와 transcript.md 사전 생성

회의를 시작하기 **직전에** 산출 폴더와 빈 transcript.md를 먼저 만든다.

```
<원본폴더>/<원본명>-검토-<YYYYMMDD-HHMM>/회의록-transcript.md
```

타임스탬프는 `date +%Y%m%d-%H%M`으로 얻는다 (Step 4에서 같은 폴더를 그대로
재사용한다 — 새로 만들지 말 것).

transcript.md 초기 내용(헤더만):

```markdown
# 검토 회의 회의록 — <원본 문서명>

- **대상 문서**: <원본 파일명>
- **회의 일시**: <YYYY-MM-DD HH:MM>
- **참석자**: <페르소나 이름·역할 5명 나열>
- **의장**: 진행 보조 (중립)

> 이 회의록은 회의가 진행되는 동안 실시간으로 작성된다. 각 발언이 나오는
> 즉시 이 파일에 추가된다.

---
```

### 3-1. 개회

의장 명의로 개회사를 작성·출력한다:
- 문서 제목·유형·목적 요약
- 오늘 검토의 목표
- 안건 (어떤 관점들을 다룰지)
- 참여 페르소나 소개

**개회사를 사용자 화면에 출력한 직후, `transcript.md`에 append**한다.
(`Edit` 또는 `Write`로 — 파일이 이미 존재하므로 Edit으로 헤더 뒤에 이어 쓴다.)

transcript.md 추가 형식:

```markdown
## 개회 — 의장

<개회사 전문>

---
```

### 3-2. Round 1 — 각자 의견 발표

페르소나 수만큼 `general-purpose` 서브에이전트를 **한 메시지 안에서 병렬로
spawn**한다 (동시 실행 → 앵커링 없는 독립 의견).

각 에이전트에게 보낼 프롬프트 구성. **문서·페르소나 본문을 프롬프트에 붙여넣지
말고, 파일 경로를 주고 에이전트가 직접 `Read`로 읽게 한다.** 의장이 요약·발췌해
넘기면 검토가 원문이 아닌 의장의 요약을 대상으로 이뤄져 품질이 무너진다:
```
당신은 문서 검토 회의에 참여하는 [이름], [역할/직함]입니다.

먼저 아래 두 파일을 `Read`로 **전문(원문 전체)** 읽으세요. 요약본이 아니라
처음부터 끝까지 읽어야 합니다:
1. 당신의 페르소나 브리프: <review-personas/번호-이름.md 경로>
   — 이 파일이 "당신이 누구인가"입니다. 이 인물로서 발언하세요.
2. 검토 대상 문서: <원본 문서 경로>
   — 이 문서를 처음부터 끝까지 읽으세요.

두 파일을 읽은 뒤, 당신의 관점에서 독립적으로 1차 검토하고, 회의 발언
형태로 의견을 제시하세요. 다른 참석자 의견은 아직 모릅니다.

발언에 반드시 포함할 것:
1. 종합 인상 (2~3문장)
2. 강점 2~3개
3. 우려·문제점 — 문서 내 위치(섹션/문장)를 명시해 구체적으로
4. 의문점·질문

페르소나 브리프에 묘사된 당신의 기질·가치관에 충실하게, 1인칭 발언체로
작성하세요.
```

`subagent_type`은 `general-purpose` (Read 권한이 있어 경로로 파일을 직접 읽을 수
있다). 각 에이전트가 반환하는 의견 텍스트와
**에이전트 ID/이름을 모두 보관**한다 (Round 2에서 이어가기 위함).

**Round 1 결과 출력 — 매우 중요**:
- 모든 에이전트가 응답을 반환하는 즉시(병렬 spawn이 모두 끝난 직후), 페르소나별로
  Round 1 발언을 **사용자 화면에 출력하면서 동시에 `transcript.md`에 append**한다.
- 각 페르소나 발언은 한 번에 하나씩 파일에 append (5명이면 5번의 Edit 호출).
  파일 I/O는 가볍지만, 발언이 누락 없이 기록되는지 확인하는 게 중요하다.

transcript.md 추가 형식 (각 페르소나마다):

```markdown
## Round 1 — <페르소나 이름> (<역할>)

<발언 전문 — 1인칭 발언체 그대로>

---
```

먼저 `## Round 1 — 각자 의견 발표` 같은 라운드 헤더를 한 번 append하고,
그 아래에 페르소나별 소제목으로 발언을 이어 붙인다.

### 3-3. Round 2 — 반박 및 첨언

의장으로서 Round 1 발언 전체를 취합한다. 그다음 각 페르소나 에이전트를
**`SendMessage`로 이어서** 진행한다 (자기 입장·문서 맥락 유지, 토큰 효율적).

각 에이전트에게 보낼 SendMessage 내용. **다른 사람들의 R1 발언은 요약하지 말고
반환된 발언 전문을 그대로 넣는다** (요약하면 반박할 거리가 뭉개진다). 문서·페르소나는
R1에서 이미 읽었고 컨텍스트가 유지되므로 다시 넘기지 않는다:
```
Round 1이 끝났습니다. 다른 참석자들의 의견은 다음과 같습니다:

[다른 페르소나 전원의 Round 1 발언 — 전문 그대로]

이제 Round 2입니다. 위 의견들을 듣고 다음을 발언하세요:
1. 동의·지지하는 의견 (누구의 어떤 점)
2. 반박·이견 (누구의 어떤 점에, 왜)
3. 추가 의견 (1라운드에서 못 한 말, 남의 의견이 촉발한 새 포인트)
4. 우선순위 견해 (어떤 문제가 가장 시급한가)

당신의 성향에 충실하게, 솔직하게 부딪히세요.
```

이 Round 2 호출들도 가능하면 병렬로 보낸다.

> **`SendMessage` 폴백**: `SendMessage`를 쓸 수 없는 환경이면, 각 페르소나에
> 대해 `general-purpose` 에이전트를 **새로 spawn**한다. 새 에이전트는 R1
> 컨텍스트가 없으므로 프롬프트에 ① 그 페르소나 브리프 **파일 경로**(Read로 읽게)
> ② 검토 대상 문서 **파일 경로**(Read로 읽게) ③ 그 페르소나의 Round 1 발언 전문
> ④ 다른 전원의 Round 1 발언 전문 ⑤ 위 Round 2 지시사항을 담는다. 여기서도
> 문서·페르소나 본문은 붙여넣지 말고 경로로 직접 Read하게 한다 (발언만 전문 인라인).

**Round 2 결과 출력**: Round 1과 동일하게 — 모든 응답이 돌아오면 사용자
화면 출력과 동시에 `transcript.md`에 페르소나별로 append한다.

```markdown
## Round 2 — <페르소나 이름> (<역할>)

<발언 전문>

---
```

Round 2 섹션 시작 전에 `## Round 2 — 반박 및 첨언` 라운드 헤더를 한 번
append하고, 그 아래에 페르소나별 소제목으로 이어 붙인다.

### 3-4. 합의안 도출

의장으로서 Round 1 + Round 2 전체를 종합한다:
- **공통 합의 사항**: 다수가 동의한 지점
- **쟁점·갈등**: 의견이 갈린 지점과, 가능한 절충안
- 각 항목을 **만장일치 / 다수 / 소수의견**으로 분류
- 우선순위 **High / Medium / Low** 부여

쟁점이 미해결로 남으면, 해당 페르소나 에이전트에게 `SendMessage`로 절충안
수용 여부를 **1회만** 확인한다 (무한 토론 금지). 절충 확인 발언도
`transcript.md`에 append.

최종 recommendations 목록을 의장 명의로 정리·출력하고, **`transcript.md`에도
append**한다:

```markdown
## 합의 도출 — 의장

<공통 합의 사항·쟁점·우선순위·최종 recommendations 정리>

---
```

### 3-5. 폐회

의장 명의 폐회사: 회의 요약, 핵심 결론, 다음 단계(산출물 안내).

**폐회사도 `transcript.md`에 append**한다:

```markdown
## 폐회 — 의장

<폐회사 전문>
```

이 시점에서 `transcript.md`는 사실상 완성된다.

## Step 4 — 나머지 산출물 생성

산출 폴더와 `회의록-transcript.md`는 **Step 3-0에서 이미 만들었고 회의 진행
중 실시간으로 채워졌다.** Step 4에서는 산출 폴더 안에 **나머지 2개 파일**을
작성한다. 폴더는 그대로 재사용한다 — 새 폴더를 만들지 말 것.

형식은 `references/output-templates.md`를 정확히 따른다.

1. **`회의록-transcript.md`** — 이미 회의 중에 실시간으로 작성됨. Step 4에서는
   파일이 제대로 끝났는지(폐회까지 들어갔는지) 점검만 한다. 추가 편집 불요.

2. **`recommendations.md`** — 우선순위순 합의 추천안. 각 항목:
   이슈 · 추천 내용 · 우선순위 · 제기한 페르소나 · 합의 수준 · 근거.

3. **`<원본명>-v2.<확장자>`** — 원본에 **합의·다수 recommendations를 반영**한
   개선 문서.
   - 원본의 의도·구조·톤을 유지하고, 추천된 변경만 반영한다.
   - **소수의견은 반영하지 않되**, 문서 하단 "미반영 의견" 섹션에 기록한다.
   - 문서 상단에 **변경 이력(changelog)**: 무엇을 왜 바꿨는지.
   - `<원본명>-v2`가 이미 존재하면 `-v3`, `-v4`로 번호를 올린다 (`Glob`으로 확인).

> **원본 보호**: 원본 파일은 어떤 경우에도 수정·삭제하지 않는다. V2는 항상
> 별도 파일이다.

작성 완료 후 사용자에게 3개 파일의 경로를 정리해 보고한다.

## 핵심 원칙

- **의장은 중립**: 본인 검토 의견을 내지 마라. 모든 내용 판단은 페르소나가 한다.
- **페르소나는 매번 새로**: 아키타입에서 고르지 말고 이 문서에서 도출해 새로
  만든다. 각 페르소나는 `review-personas/` 폴더에 파일로 남긴다.
- **페르소나 ≠ 검토 결론**: 페르소나 파일은 "어떤 사람인가"(경력·기질·
  가치관·말투)만 담는다. "이 문서의 무엇이 문제다" 같은 검토 의견은 절대
  미리 적지 않는다 — 그것은 Step 3 회의에서 인물이 직접 만들어 낸다.
- **원문 직접 읽기**: 의장은 문서·페르소나를 요약·발췌해 에이전트에게 넘기지
  않는다. 에이전트가 원본 파일과 자기 페르소나 파일을 **경로로 직접 `Read`**한다.
  의장이 인라인으로 전달하는 것은 Round 2의 "다른 사람 발언"뿐이며, 그것도
  요약 없이 전문 그대로 넘긴다.
- **진짜 독립성**: Round 1은 반드시 병렬 spawn — 서로의 의견을 모른 채 시작.
- **갈등을 살려라**: 페르소나끼리 의견이 부딪히는 게 정상이고 가치다. 억지로
  봉합하지 말고, 합의 못 한 것은 "소수의견"으로 정직하게 남겨라.
- **구체성**: 모든 지적은 문서 내 위치를 명시해야 한다. "전반적으로 약하다"
  같은 막연한 말은 페르소나에게 다시 요청한다.
- **원본 불가침**: V1은 절대 건드리지 않는다.
- **실시간 기록**: 회의가 시작될 때 `transcript.md`를 먼저 만들고, 발언이
  나올 때마다 즉시 append한다. 회의가 중간에 끊겨도 그때까지의 회의록은
  파일에 남아 있어야 한다. 회의 끝나고 한 번에 쓰는 방식은 금지.
- 회의·산출물의 언어는 원본 문서의 언어를 따른다 (한국어 문서 → 한국어 회의).
